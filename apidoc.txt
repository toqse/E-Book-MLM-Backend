MLM Platform — API Documentation
==================================
Base URL (local): http://localhost:8000
API prefix: /api/v1/
Timezone: Asia/Kolkata (IST); timestamps ISO 8601 in responses where applicable.

CONVENTIONS
-----------
Authentication: Bearer JWT in header:
  Authorization: Bearer <access_token>

Standard JSON envelope (most endpoints):
  Success:
    { "success": true, "data": { ... }, "message": "<string>", "errors": null }
  Error:
    { "success": false, "data": null, "message": "<string>", "errors": { ... } }

Pagination (list endpoints returning results):
  { "success": true, "data": { "count": N, "next": null, "previous": null, "results": [ ... ] }, ... }
  (Some list endpoints return a custom { "results": [...], "count": ... } inside data.)

Monetary amounts: returned as strings with two decimals where applicable (e.g. "30.00").

Roles (admin staff users): SUPER_ADMIN, FINANCE, SUPPORT, MEMBER.
Admin APIs require is_staff=True and appropriate role (see each endpoint).

Throttling: default DRF throttles — authenticated users ~100/min (see REST_FRAMEWORK in settings).

------------------------------------------------------------------------------
FLOWS (high level)
------------------------------------------------------------------------------

1) Member registration (OTP + JWT)
   - POST /api/v1/auth/send-otp/  purpose=REGISTER, phone or email
   - POST /api/v1/auth/verify-otp-register/  otp_code, full_name, optional referral_code
   - Response includes access + refresh JWT and profile stub.
   - Referral: optional ?ref= on marketing site is sent as referral_code; sponsor FK is immutable after registration.

2) Member login
   - POST /api/v1/auth/send-otp/  purpose=LOGIN
   - POST /api/v1/auth/verify-otp-login/

3) Token refresh / logout
   - POST /api/v1/auth/token/refresh/  (SimpleJWT; body: { "refresh": "<token>" })
   - POST /api/v1/auth/logout/  body: { "refresh": "<token>" }  (blacklists refresh token)

4) KYC and payout details
   - POST /api/v1/auth/kyc/submit/  PAN + Aadhaar (Aadhaar stored masked server-side)
   - GET  /api/v1/auth/kyc/status/
   - POST /api/v1/auth/bank/  bank account / UPI / preference

5) Course catalog and enrollment
   - GET /api/v1/courses/  (public) — active e-books
   - GET /api/v1/courses/<slug>/  (public) — detail
   - Purchase:
     - POST /api/v1/payments/create-order/
       Fields: ebook_slug (optional), sponsor_code (optional sponsor-slot code), is_retail (optional bool)
     - If valid sponsor slot: amount due 0; GST invoice still generated server-side.
     - Else Razorpay order created; client pays via Razorpay Checkout.
     - POST /api/v1/payments/verify/  razorpay_payment_id, razorpay_signature, order_id
   - GET /api/v1/user/courses/enrolled/
   - GET /api/v1/user/courses/<slug>/download/ — returns URL + expiry metadata (integrate Azure SAS in production)

6) Binary tree placement (after first qualifying MLM purchase)
   - On successful PAID order (non-retail): user becomes member, BinaryTreeService places node under sponsor subtree (BFS, left-first).
   - GET /api/v1/user/tree/ — own subtree snapshot (limited depth)
   - GET /api/v1/user/tree/uplines/ — ancestor chain
   - GET /api/v1/user/tree/downlines/ — downline list (simplified listing)

7) Commission engine (async via Celery in production; eager in tests)
   - On PAID MLM order: direct ₹30 to sponsor; up to three binary uplines × ₹10 each, skipping sponsor duplicate (one-credit rule).
   - Earning cap from SystemConfig (default ₹22,200) enforced per recipient.
   - Milestone bonuses on sponsor direct-referral milestones (simplified table in engine).
   - Refund within window: commissions reversed; enrollment removed.

8) Wallet, bands, withdrawals
   - Wallet balances updated on commission credit/debit.
   - Crossing configured band thresholds may auto-issue sponsor-slot batches (see wallet/bands.py).
   - Member requests withdrawal per band; finance admin approves (Razorpay Payout integration to be wired with env keys).

9) Sponsor slots
   - Slot codes issued on slot-type bands; validate publicly at checkout.
   - Member lists batches/codes; share channel logged.

10) Admin / back-office
   - Admin login: password + OTP 2FA pattern (endpoints provided).
   - Dashboards, user/KYC management, commissions, withdrawals, reports (stubs where noted in code).

ENDPOINT CATALOG
------------------------------------------------------------------------------

MODULE: Authentication (/api/v1/auth/)
POST   /api/v1/auth/send-otp/
       Body: { "phone": "<optional>", "email": "<optional>", "purpose": "REGISTER"|"LOGIN"|"KYC"|"ADMIN_LOGIN" }
       Public. Rate limit: max 3 OTP sends / 10 minutes per identifier (cache counter).

POST   /api/v1/auth/verify-otp-register/
       Body: { "phone"|"email", "otp_code", "full_name", "referral_code"?: str }
       Public. Creates User with member_id / referral_code / referral_link.

POST   /api/v1/auth/verify-otp-login/
       Body: { "phone"|"email", "otp_code" }
       Public.

       Body: { "refresh": "<refresh_token>" }  — SimpleJWT.

       Auth: Bearer. Body: { "refresh": "<refresh_token>" } — blacklists refresh token.

INFRASTRUCTURE & SETTINGS
------------------------------------------------------------------------------
- Production database: MySQL 8.x (see config/settings/production.py and .env.example).
- Tests / CI: SQLite in-memory (config/settings/testing.py); run pytest from project root:
    set DJANGO_SETTINGS_MODULE=config.settings.testing
    pytest
- Celery + Redis: broker/result backend from REDIS_URL; beat schedule includes daily sponsor-slot expiry task.
- Docker: see mlm_backend/docker/docker-compose.yml (web, nginx, mysql, redis, celery, celery-beat).
