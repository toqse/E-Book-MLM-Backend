MLM Platform — API Documentation
==================================
Base URL (local): http://localhost:8000
API prefix: /api/v1/
Timezone: Asia/Kolkata (IST); timestamps ISO 8601 in responses where applicable.

CONVENTIONS
-----------
Authentication (member/admin JWT): Bearer in header:
  Authorization: Bearer <access_token>

Standard JSON envelope (most endpoints — see exceptions below):
  Success:
    { "success": true, "data": { ... }, "message": "<string>", "errors": null }
  Error:
    { "success": false, "data": null, "message": "<string>", "errors": <object|null> }

Exceptions to the envelope:
  • CSV export endpoints — return plain text/csv HTTP responses with Content-Disposition;
    Content-Type: text/csv.

Validation errors may use DRF’s field error structure inside "errors" (via envelope
exception handler) for 400-class failures.

Pagination: many list endpoints return { "results": [...], "count": ... } inside "data".

Monetary amounts: returned as decimal strings where applicable (e.g. "30.00", "0").

Staff roles (is_staff=True; User.role): SUPER_ADMIN, FINANCE, SUPPORT. MEMBERS are not staff.
Permission classes in code:
  • IsAdminRole — SUPER_ADMIN, FINANCE, or SUPPORT + is_staff (broad admin).
  • IsFinanceAdmin — SUPER_ADMIN or FINANCE + is_staff.
  • IsSupportAdmin — SUPER_ADMIN or SUPPORT + is_staff.
  • IsSuperAdmin — SUPER_ADMIN only.

Throttling (REST_FRAMEWORK): user ~100/min, anon ~30/min unless overridden.

-----------------------------------------------------------------------------
FLOWS (high level)
-----------------------------------------------------------------------------

1) Member registration (OTP + JWT)
   - POST /api/v1/auth/register/send-otp/  — mandatory: phone (E.164 e.g. +918589960592), full_name,
     referral_code; optional: email. Validates sponsor; stores profile on the OTP server-side.
   - POST /api/v1/auth/verify-otp-register/  — body: phone (same E.164) + otp_code only (completes signup).
   - Generic POST /api/v1/auth/send-otp/ does NOT accept purpose=REGISTER (use register/send-otp/).
   - LOGIN / KYC OTP (phone channel): same strict E.164 with leading + as registration (e.g. +918589960592).
   - data.tokens + data.user on successful verify.
   - Company referral code "Admin": see DEFAULT_COMPANY_REFERRAL_CODE / superuser (earlier docs).

2) Member login
   - POST /api/v1/auth/send-otp/  purpose=LOGIN
   - POST /api/v1/auth/verify-otp-login/

3) JWT (access only) / logout
   - Access token TTL: JWT_ACCESS_TOKEN_LIFETIME_MINUTES (default 60) in django-rest-framework-simplejwt.
   - No refresh token endpoint; login/register again after expiry for a new token.
   - POST /api/v1/auth/logout/ — Bearer optional; discard access token client-side after call.

4) Compliance (agreements OTP + extended KYC/bank/nominee)
   - GET /api/v1/agreements/ (Bearer) — active legal documents (content_html + PDF URL when present); optional category filter.
   - POST /api/v1/agreements/send-otp/ — JSON { "document_ids": [int, ...] }; OTP tied to JWT user identity only.
   - POST /api/v1/agreements/verify/ — JSON { "document_ids", "otp_code" }; records UserAgreementAcceptance + client IP.
  - POST /api/v1/auth/compliance/submit/ — multipart form: identity, address, nominee, bank, payout; files pan_document,
    aadhar_front and aadhar_back (both required on first submit; optional on update if already stored). Dates in DD/MM/YYYY.
     Requires all active LegalDocuments with requires_acceptance_for_compliance=true at current version.
   - GET /api/v1/auth/kyc/status/ — kyc_status, timestamps, missing_acceptances, has_compliance_profile, etc.
   - Deprecated (410 Gone): POST /api/v1/auth/kyc/submit/, POST /api/v1/auth/bank/ — use compliance/submit/.

5) Course catalog and enrollment
   - GET /api/v1/courses/ (public) — grouped catalog without pdf_url; with Bearer, each book includes is_already_purchased; filters and pagination unchanged
   - GET /api/v1/courses/<slug>/ (public)
   - Cart (Bearer): GET /api/v1/user/cart/ — items + totals preview; POST /api/v1/user/cart/items/ — body ebook_id or ebook_slug; DELETE /api/v1/user/cart/items/<id>/; DELETE /api/v1/user/cart/ — clear; POST /api/v1/user/cart/checkout/ — same billing/sponsor/is_retail as create-order, response shape matches create-order (then POST /api/v1/payments/verify/ with order_id)
   - Purchase (direct buy): POST /api/v1/payments/create-order/ (Bearer) — ebook_id or ebook_slug (optional; both → id wins), sponsor_code / sponsor_slot_code, is_retail, optional billing_* snapshot fields for the GST invoice
   - Zero-amount path when valid sponsor slot; else Razorpay order (amount_paise, razorpay_order_id, key_id).
   - POST /api/v1/payments/verify/ (Bearer) — order_id + Razorpay payment id/signature; on success order is PAID, enrollment + GST invoice PDF generated when applicable
   - GET /api/v1/user/orders/ (Bearer) — recent orders with amount_paid; PAID rows may include pdf_url (absolute URL to invoice PDF when available)
   - GET /api/v1/user/orders/<id>/invoice/ (Bearer) — invoice_number + pdf_url for that order
   - GET /api/v1/user/courses/enrolled/ — same grouped shape as GET /courses/ for published books you are enrolled in; optional category/language/search and pagination
   - GET /api/v1/user/courses/enrolled/<slug>/ | /enrolled/<id>/ — same body as GET /courses/<slug>|<id>/ (pdf_url + is_already_purchased); 403 if not enrolled
   - GET /api/v1/user/courses/<slug>/download/ — URL + expires_at (SAS in production)

6) Binary tree (after qualifying MLM purchase)
   - GET /api/v1/user/tree/, /subtree/, /uplines/, /downlines/, /level/<n>/

7) Commission engine (Celery in prod; eager in tests)
   - Direct + upline + milestones; cap from SystemConfig; refunds reverse commissions.

8) Wallet, bands, withdrawals
   - Prefer GET /api/v1/user/earnings/ and GET /api/v1/user/payouts/ for dashboard bundles.
   - Band ladder unlocks sponsor-slot batches at configured indices.
   - Member POST withdraw; FINANCE approve/reject/export.

9) Sponsor slots — validate publicly; member lists batches; share logs channel.

10) Admin — password + OTP patterns; dashboards, users, reports (some stubs).

DETAILED ENDPOINT REFERENCE
=============================================================================

Every subsection lists: METHOD URL | Auth | Request | Success data (inside envelope
unless noted) | Common errors.


MODULE: Authentication (/api/v1/auth/)
--------------------------------------------------------------------------------
POST /api/v1/auth/send-otp/
  Auth: None.
  Body (JSON): {
    "phone": "<optional when using email login; strict E.164 with + when supplied — e.g. +918589960592>",
    "email": "<optional>",
    "purpose": "LOGIN" | "KYC" | "ADMIN_LOGIN"
  }
  Rules: Provide phone (E.164 +…) and/or email. Bare domestic numbers without + are rejected. Not used for signup.
  Success data (when EXPOSE_OTP_IN_RESPONSE=true, default unless env disables):
    {
      "expires_in_seconds": 600,
      "otp": "<six-digit code; also INFO-logged under logger apps.authentication>"
    }
    When EXPOSE_OTP_IN_RESPONSE=false, "otp" is omitted (production SMS flows).
  message: "OTP sent"
  Errors: 429 — rate limited (max 3 sends / 10 minutes per identifier);
    errors may include {"detail": "rate_limited"}.

POST /api/v1/auth/register/send-otp/
  Auth: None.
  Body: {
    "phone": "<required; strict E.164 with leading +: e.g. +918589960592>",
    "email": "<optional; stored if provided>",
    "full_name": "<required>",
    "referral_code": "<required; e.g. Admin for company sponsor>"
  }
  Validates sponsor exists; rejects duplicate phone (or duplicate email when email supplied).
  Success: same envelope as send-otp (expires_in_seconds + otp when EXPOSE_OTP_IN_RESPONSE is on).
  Errors: 400 invalid referral / validation / duplicate user; 429 rate limit per phone.

POST /api/v1/auth/verify-otp-register/
  Auth: None.
  Body: {
    "phone": "<required; same E.164 as register/send-otp>",
    "otp_code": "6 chars"
  }
  Creates user from data stored on the matching unused REGISTER OTP (full_name, optional email, sponsor).
  Success data: {
    "user": { id, member_id, full_name, phone, email, referral_code, is_member, role },
    "tokens": { "access" }
  }
  Errors: 400 invalid OTP / expired session data / duplicate phone-email; note: complete within OTP TTL after send.

POST /api/v1/auth/verify-otp-login/
  Auth: None.
  Body (JSON): identifier — phone (strict E.164 with +, same as send-otp) or email — plus otp_code (6 chars).
  Success data: same user + tokens shape as register.
  Errors: 423 account locked (otp_locked_until); 404 user not found; 400 OTP errors.

POST /api/v1/auth/logout/
  Auth: Bearer access JWT recommended (session context); body optional.
  Success: data null; message "Logged out" — client removes stored access JWT.

GET/PATCH /api/v1/auth/me/
  Auth: Bearer.
  GET — Success data: same user stub as verify (id, member_id, full_name, phone, email,
    referral_code, is_member, role).
  PATCH — Body may include full_name and/or payout_preference (passed through to User).
  Success data: updated user stub; message "Updated"

POST /api/v1/auth/kyc/submit/
  Deprecated: returns 410 Gone — use POST /api/v1/auth/compliance/submit/.

GET /api/v1/auth/kyc/status/
  Auth: Bearer.
  Success data includes: kyc_status, kyc_submitted_at, kyc_reviewed_at, kyc_rejection_reason (or null),
    pan_last4, missing_acceptances (list of docs needing acceptance at latest version),
    has_compliance_profile, compliance_submission_version,
    compliance_submission (null if no profile, else: personal_details from User + address/DOB/gender;
    kyc_details (PAN/Aadhaar fields + verification_documents URLs); bank_details; nominee_details; updated_at).

POST /api/v1/auth/bank/
  Deprecated: returns 410 Gone — use POST /api/v1/auth/compliance/submit/.

POST /api/v1/auth/compliance/submit/
  Auth: Bearer. Content-Type: multipart/form-data (or multipart with JSON-compatible fields supported by parser).
  Text fields:
    date_of_birth (DD/MM/YYYY), gender (M|F|O|U),
    full_address, city, pin_code, state, country,
    pan_number, name_on_pan, aadhar_number (12 digits), name_on_aadhar,
    nominee_name, nominee_relationship, nominee_phone (strict E.164 +91 Indian mobile rule),
    nominee_date_of_birth (DD/MM/YYYY),
    account_holder_name, account_number, bank_name, ifsc, branch,
    account_type (SAVINGS|CURRENT|NRO|NRE|RECURRING|OTHER),
    optional payout_preference (BANK|UPI); omitted or blank defaults to UPI,
    optional upi_id for UPI payouts.
  Files: pan_document, aadhar_front, aadhar_back (PDF or image types as accepted by storage).
  Server syncs PAN + masked Aadhaar onto User alongside MemberComplianceProfile; sets kyc_status PENDING pending admin review.
  Success: { "kyc_status", "compliance_submission_version" }; 400 if mandatory agreements missing.

POST /api/v1/auth/validate-referral/
  Auth: None.
  Body: { "referral_code": "CODE" } or { "code": "CODE" }
  Success data: { "sponsor_name": "<full_name>" }
  Errors: 404 invalid code.

MODULE: User — Binary tree (/api/v1/user/tree*)
--------------------------------------------------------------------------------
GET /api/v1/user/tree/
  Auth: Bearer.
  Success data: If user has no binary node: { "root": null }.
  Else { "root": nested tree depth ≤ 3 }: each node has member_id, full_name,
    initials, kyc_status, account_status, is_active, rank_label (null),
    position, level, left, right (null if absent).

GET /api/v1/user/tree/subtree/
  Auth: Bearer. Drill-down: nested subtree under an anchor member (same node shape
    as GET /api/v1/user/tree/; batched load, enriched fields as on /user/tree/).
  Query: anchor_member_id (string, optional). Omit or use your own member_id to
    anchor on yourself. Otherwise must be a member_id in your binary downline
    (including yourself when placed).
  Query: max_depth (int, optional, default 3, max 10). Rings below the anchor;
    same semantics as the nested tree builder (depth 0 = anchor).
  Success data: { "root": nested | null, "anchor_member_id", "max_depth" }.
    root null when anchor is you but you have no binary node, or when anchor is
    authorized but not placed (defensive).
  Errors: 404 if anchor_member_id does not match any member; 403 if anchor is
    not yourself or in your binary subtree; 403 if you have no binary node and
    you request another member's subtree.

GET /api/v1/user/tree/uplines/
  Success data: { "uplines": [ { member_id, full_name, level }, ... ] } up to 4 ancestors.

GET /api/v1/user/tree/downlines/
  Success data: { "results": [ { member_id, full_name, level } ], "count": <int> } BFS.

GET /api/v1/user/tree/level/<n>/
  Auth: Bearer (n integer in path).
  Success data (stub): { "level": n, "members": [] }

POST /api/v1/user/tree/place-direct/
  Auth: Bearer. Caller must be the direct sponsor of the buyer.
  Body: { "member_id": "<buyer member_id>", "leg": "LEFT" | "RIGHT" } (alias: position).
  Rules: buyer has a PAID non-retail order in PENDING (or FAILED after auto error)
    placement queue, no binary_node yet, and current time is on or before
    placement_deadline_at while status is PENDING (FAILED allows sponsor recovery
    without deadline).
  Success: { "member_id", "leg", "order_id" } — binary placed; commissions run
    for all eligible PAID MLM orders for that buyer (idempotent per order).
  Errors: 400 validation; 404 member; 400 no pending queue / wrong sponsor /
    window expired.

MODULE: Admin — Binary placement queue
--------------------------------------------------------------------------------
GET /api/v1/admin/placements/pending/
  Auth: Bearer + IsAdminRole.
  Success: {
    "results": [
      { "order_id", "order_number", "buyer_member_id", "buyer_name",
        "sponsor_member_id"|null, "placement_deadline_at", "placement_status" }
    ],
    "count": <int>
  }
  Lists PAID MLM orders awaiting or recovering placement (PENDING or FAILED).

POST /api/v1/admin/placements/<order_id>/reverse/
  Auth: Bearer + IsAdminRole.
  Leaf-only: buyer must have a binary node with no left/right children.
  Reverses all credited commissions on that buyer’s PAID MLM orders, removes
    the binary node, resets this order’s placement to PENDING with a new
    deadline from platform settings.
  Success: { "order_id", "status": "reversed" }. 409 if not leaf.

POST /api/v1/admin/placements/<order_id>/reassign/
  Auth: Bearer + IsAdminRole.
  Body: either "leg": "LEFT"|"RIGHT" (manual under sponsor) or "strategy":
    "LEFT_FIRST"|"RIGHT_FIRST"|"LONG_LEG"|"WEAK_LEG" (auto under sponsor).
  If buyer already placed: reverse (leaf-only) then place again with
    placement_status PLACED_ADMIN.
  Success: { "order_id", "status": "reassigned" }.


MODULE: User — Earnings bundle (preferred for My Earnings UI)
--------------------------------------------------------------------------------
GET /api/v1/user/earnings/
  Auth: Bearer.
  Query: include — optional. Comma-separated: "overview" (default when omitted),
    "ledger", or "overview,ledger" or "all" (both). Use "ledger" only to skip overview.
  Query: period — optional: today | 7d | 30d | fy | all (default all). Filters ledger.
  Query: type — optional: all | direct | passive | milestone | reversed | pending.
  Query: page (default 1), page_size (default 20, max 100) for ledger section.
  Success data (subset keys omitted when include excludes that section):
    "overview": {
      "lifetime_total", "breakdown" { direct, passive, milestone, slots } each with
        amount string and unit hints (units, unit_amount, codes_redeemed for slots),
      "wallet" { cash_balance, total_earned, total_withdrawn, on_hold, tds_fy, reversed },
      "cap" { limit, used, used_percent, remaining },
      "kyc" { status, withdrawals_blocked },
      "rates" { direct_commission, upline_commission, earning_cap,
        is_repurchase_commission_allowed (bool, mirrors SystemConfig; default false) },
      "fy_label"
    },
    "ledger": {
      "results": [ unified rows ],
      "total_count", "page", "page_size"
    }
  Ledger row fields: id, kind (COMMISSION|MILESTONE), at, display_type, level,
    description, triggered_by { member_id, full_name }|null, via_downline, gross,
    tds, net, status, order_id.
  Notes: on_hold = sum of credited commission net still under Order.refund_eligible_until.
    Per-line tds may be 0 until engine writes tds_deducted. Slots amount =
    count(REDEEMED sponsor codes issued to you) × ₹100 display unit.

MODULE: Payments & orders
--------------------------------------------------------------------------------
POST /api/v1/payments/create-order/
  Auth: Bearer (member JWT).
  Content-Type: application/json.
  Body: {
    "ebook_id": "<optional integer; if both ebook_id and ebook_slug are sent, ebook_id wins>",
    "ebook_slug": "<optional string; must reference a published book>",
    "sponsor_code": "<optional; alias for redeeming sponsor slot>",
    "sponsor_slot_code": "<optional; same semantics as sponsor_code if sponsor_code omitted>",
    "is_retail": <optional boolean, default false — retail purchase vs MLM path>,
    "billing_line1": "<optional string; max 255; stored on order for the invoice 'Invoice To' address block>",
    "billing_line2": "<optional string; max 255>",
    "billing_city": "<optional string; max 128>",
    "billing_state": "<optional string; max 128>",
    "billing_postal_code": "<optional string; max 20>",
    "billing_country": "<optional string; max 128>"
  }
  Rules:
    • Omitting both ebook_id and ebook_slug selects the configured primary published e-book (if any).
    • Billing fields are optional for API compatibility; when sent, values are trimmed and length-capped
      server-side and snapshotted on the Order (they appear on the generated GST invoice PDF).
  Success data:
    • Paid checkout (Razorpay): {
        "order_id", "order_number", "amount_paise" <int>,
        "razorpay_order_id", "key_id" (publisher key for client SDK)
      }
    • Zero-amount (valid sponsor slot): {
        "order_id", "order_number", "amount_paise": 0,
        "razorpay_order_id": null, "key_id", "status" (typically PAID; enrollment finalized server-side)
      }
  Errors:
    • 401 — missing/invalid Bearer token.
    • 404 — book not found / not published when ebook_id or ebook_slug was explicitly provided.
    • 400 — when ebook_id or ebook_slug was provided: user already has an enrollment for that book.
    • 500 — Razorpay not configured, selected book unpublished, gateway errors, etc. (message in envelope).

POST /api/v1/payments/verify/
  Auth: Bearer (member JWT; order must belong to the authenticated user).
  Content-Type: application/json.
  Body: {
    "order_id": <int>,
    "razorpay_payment_id": "<string>",
    "razorpay_signature": "<string>"
  }
  Success data: { "status": "PAID", "order_number" }
  Side effects when transitioning to PAID (same pattern as manual admin verify):
    enrollment for the purchased e-book, MLM placement queue when applicable,
    GST invoice row + invoice PDF attachment (generated server-side; see pdf_url below).
  Errors:
    • 401 — not authenticated.
    • 404 — order_id not found for this user.
    • 400 — Razorpay signature mismatch / invalid payload / gateway error (message in envelope).
  MLM (non-retail): first PAID order for a member not yet in the binary tree
    sets placement_status=PENDING and placement_deadline_at (hours from
    placement_manual_window_hours). Sponsor places via POST .../tree/place-direct/;
    after the deadline Celery auto-places using auto_placement_strategy.
    Commissions credit only after binary placement succeeds (idempotent per order).

POST /api/v1/payments/webhook/
  Auth: None.
  Header: X-Razorpay-Signature — HMAC verified against raw body (see settings.RAZORPAY_KEY_SECRET).
  Body: Razorpay JSON payload (raw bytes used for signature).
  Success data: { "received": true, "event": "<from payload>" }
  Errors: 400 — bad signature.
  Note: handler currently acknowledges receipt; client flow still relies on POST .../payments/verify/ after pay.

GET /api/v1/user/orders/
  Auth: Bearer.
  Success data: {
    "results": [
      {
        "id": <int>,
        "order_number": "<string>",
        "status": "CREATED" | "PAID" | "FAILED" | "REFUNDED",
        "purchased_at": "<ISO datetime string|null>",
        "ebook_title": "<string|null>",
        "thumbnail_url": "<absolute HTTPS/HTTP URL string|null>",
        "amount_paid": "<decimal string>",
        "pdf_url": "<absolute HTTPS/HTTP URL string|null>"
      },
      ... newest first, max 50
    ]
  }
  pdf_url:
    • null for non-PAID orders, or when no GST invoice exists yet.
    • For PAID orders: absolute URL to the invoice PDF file when generation succeeded (server may generate
      on demand if missing). null if PDF file is still unavailable (e.g. generation error).
    • Clients must serve/open media URLs per deployment (e.g. SERVE_MEDIA=1 locally, CDN/nginx in prod).

GET /api/v1/user/orders/<id>/invoice/
  Auth: Bearer; <id> is the order primary key for the authenticated user’s order.
  Success data: {
    "invoice_number": "<string>",
    "pdf_url": "<absolute URL|null — prefers FileField-backed invoice PDF; legacy DB pdf_url column if no file>"
  }
  Errors: 404 — order not found, not yours, or no GST invoice yet.

Invoice PDF branding (operators): optional environment variables COMPANY_NAME, GST_NUMBER,
  COMPANY_ADDRESS, COMPANY_PHONE, COMPANY_EMAIL, COMPANY_WEBSITE,
  INVOICE_PAYMENT_DETAILS, INVOICE_TERMS_AND_CONDITIONS (see mlm_backend/.env.example).

POST /api/v1/user/orders/<id>/refund/
  Auth: Bearer; <id> must be your order.
  Allowed only when status is PAID and now <= refund_eligible_until (from SystemConfig.refund_window_days).
  Success data: { "status": "REFUNDED" } — reverses commissions (when applicable), removes enrollment tied to this order.
  Errors: 401; 400 — not PAID / refund window expired; 404 — order not found or not yours.

GET /api/v1/admin/orders/
  Auth: IsFinanceAdmin.
  Success: { "results": [ { "id", "status" }, ... ] }

POST /api/v1/admin/orders/<order_ref>/verify-payment-manual/
  Path `order_ref`: numeric **database id** (e.g. `6`) **or** the string **order_number**
    (e.g. `ORD-20260504-E3473F`). Using only the order number in the URL avoids 404 when
    the route previously required `<int:...>`.
  Auth: Bearer + IsSuperAdmin only (same gate as system config).
  Body (optional JSON): { "note": "reason for audit log" }.
  Effect: same as successful Razorpay POST /api/v1/payments/verify/ for that order:
    status PAID, paid_at, refund_eligible_until, razorpay_payment_id set to a
    traceable MANUAL-<admin_user_id>-<hex> value (max 64 chars), enrollment,
    placement queue for MLM, GST invoice when applicable.
  Only allowed when order status is CREATED (400 if REFUNDED / other).
  Idempotent: if order is already PAID, returns 200 with message that it was
    already paid (no duplicate audit).
  Success: { "status": "PAID", "order_number", "order_id" }; writes audit action
    payment.admin_verified on first transition to paid.
  Errors: 404 order; 400 status message; 403 non–super-admin.
