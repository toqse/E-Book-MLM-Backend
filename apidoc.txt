MLM Platform — API Documentation
==================================
Base URL (local): http://localhost:8000
API prefix: /api/v1/
Timezone: Asia/Kolkata (IST); timestamps ISO 8601 in responses where applicable.

CONVENTIONS
-----------
Authentication (member/admin JWT): Bearer in header:
  Authorization: Bearer <access_token>

Standard JSON envelope (most endpoints — see exceptions below):
  Success:
    { "success": true, "data": { ... }, "message": "<string>", "errors": null }
  Error:
    { "success": false, "data": null, "message": "<string>", "errors": <object|null> }

Exceptions to the envelope:
  • CSV export endpoints — return plain text/csv HTTP responses with Content-Disposition;
    Content-Type: text/csv.
  • Some admin exports (e.g. GET /api/v1/admin/commissions/export/ with export_format=pdf) return
    application/pdf with Content-Disposition: attachment.

Validation errors may use DRF’s field error structure inside "errors" (via envelope
exception handler) for 400-class failures.

Pagination: many list endpoints return { "results": [...], "count": ... } inside "data".

Monetary amounts: returned as decimal strings where applicable (e.g. "30.00", "0").

Staff roles (is_staff=True; User.role): SUPER_ADMIN, FINANCE, SUPPORT. MEMBERS are not staff.
Permission classes in code:
  • IsAdminRole — SUPER_ADMIN, FINANCE, or SUPPORT + is_staff (broad admin).
  • IsFinanceAdmin — SUPER_ADMIN or FINANCE + is_staff.
  • IsSupportAdmin — SUPER_ADMIN or SUPPORT + is_staff.
  • IsSuperAdmin — SUPER_ADMIN only.

Throttling (REST_FRAMEWORK): user ~100/min, anon ~30/min unless overridden.

-----------------------------------------------------------------------------
FLOWS (high level)
-----------------------------------------------------------------------------

1) Member registration (OTP + JWT)
   - POST /api/v1/auth/register/send-otp/  — mandatory: phone (E.164 e.g. +918589960592), full_name,
     referral_code; optional: email. Validates sponsor; stores profile on the OTP server-side.
   - POST /api/v1/auth/verify-otp-register/  — body: phone (same E.164) + otp_code only (completes signup).
   - Generic POST /api/v1/auth/send-otp/ does NOT accept purpose=REGISTER (use register/send-otp/).
   - LOGIN / KYC OTP (phone channel): same strict E.164 with leading + as registration (e.g. +918589960592).
   - data.tokens + data.user on successful verify.
   - Company referral code (default "Admin"): env DEFAULT_COMPANY_REFERRAL_CODE, overridable via
     SystemConfig (PATCH superadmin GET/PATCH /api/v1/admin/config/ or PATCH /api/v1/auth/me/); see auth/me admin block.

2) Member login
   - POST /api/v1/auth/send-otp/  purpose=LOGIN
   - POST /api/v1/auth/verify-otp-login/

3) JWT (access only) / logout
   - Access token TTL: JWT_ACCESS_TOKEN_LIFETIME_MINUTES (default 60) in django-rest-framework-simplejwt.
   - No refresh token endpoint; login/register again after expiry for a new token.
   - POST /api/v1/auth/logout/ — Bearer optional; discard access token client-side after call.
   - Suspended/deactivated accounts: Bearer access tokens are rejected while the account is suspended (tokens become valid again after unsuspend).

4) Compliance (agreements OTP + extended KYC/bank/nominee). GET /api/v1/auth/me/ redacts referral_code,
   referral_link, tax_withholding, withdrawal_band, earning_cap, binary_placement, and team_legs until KYC is
   VERIFIED with a compliance profile on file.
   - GET /api/v1/agreements/ (Bearer) — active legal documents + `user` array (order invoice URLs, compliance acceptance proof download URL); optional category filter.
   - GET /api/v1/agreements/acceptance-proof/<uuid>/download/ — download agreement acceptance proof PDF (attachment). Use full `pdf_download_url` from agreements (includes signed `token` for browser) or Bearer.
   - GET /api/v1/agreements/compliance-legal/ (Bearer) — subset: active documents with category LEGAL DOCUMENT and requires_acceptance_for_compliance=true only; response includes `user` { id, full_name } for the authenticated member.
   - POST /api/v1/agreements/send-otp/ — JSON { "document_ids": [int, ...], "declaration": "<text>" }; OTP payload binds documents + user + declaration; declaration appears on acceptance proof PDF.
   - POST /api/v1/agreements/verify/ — JSON { "document_ids", "otp_code" }; records UserAgreementAcceptance + declaration row + client IP.
  - POST /api/v1/auth/compliance/submit/ — multipart form: identity, address, nominee, bank, payout; files pan_document,
    aadhar_front and aadhar_back (both required on first submit; optional on update if already stored). Dates in DD/MM/YYYY.
     Requires all active LegalDocuments with requires_acceptance_for_compliance=true at current version.
     KYC timing: unless SystemConfig.trigger_instant_kyc_submission is true, submission is allowed only after the refund
     window has closed on at least one PAID ebook order (403 errors.detail kyc_refund_window_active). Instant mode allows
     submit immediately after purchase (no invitation email/SMS).
   - GET /api/v1/auth/kyc/status/ — kyc_status, kyc_submission_allowed, kyc_submission_mode, timestamps, missing_acceptances, etc.
   - GET /api/v1/auth/kyc/invite/?token=... (public) — validate post-refund invitation deep link before login.
   - Deprecated (410 Gone): POST /api/v1/auth/kyc/submit/, POST /api/v1/auth/bank/ — use compliance/submit/.

5) Course catalog and enrollment
   - GET /api/v1/courses/ (public) — grouped catalog without pdf_url; with Bearer, each book includes is_already_purchased; filters and pagination unchanged
   - GET /api/v1/courses/<slug>/ (public)
   - Cart (Bearer): GET /api/v1/user/cart/ — items + totals preview; POST /api/v1/user/cart/items/ — body ebook_id or ebook_slug; DELETE /api/v1/user/cart/items/<id>/; DELETE /api/v1/user/cart/ — clear; POST /api/v1/user/cart/checkout/ — same billing/sponsor/is_retail as create-order; response = create-order fields + full_name + phone for the buyer (then POST /api/v1/payments/verify/ with order_id)
   - Purchase (direct buy): POST /api/v1/payments/create-order/ (Bearer) — ebook_id or ebook_slug (optional; both → id wins), sponsor_code / sponsor_slot_code, is_retail, optional billing_* snapshot fields for the GST invoice
   - Zero-amount path when valid sponsor slot; else Razorpay order (amount_paise, razorpay_order_id, key_id).
   - POST /api/v1/payments/verify/ (Bearer) — order_id + Razorpay payment id/signature; on success order is PAID, enrollment + GST invoice PDF generated when applicable; success payload: status, order_number, amount, payment_verified_at, payment_method
   - GET /api/v1/user/orders/ (Bearer) — PAID and REFUNDED orders; each `items[]` row includes **`refund_status`**: latest `RefundRequest.status` for that line (`PENDING`, `PROCESSING`, `APPROVED`, `REJECTED`) or `null` if none; includes order-wide requests (`order_line_id` null) when they apply to that line. When the latest applicable status is **REJECTED**, `items[].can_refund` is false, **`items[].refund_reject_reason`** carries the finance reject note (null otherwise), and the member cannot submit again for that line/order-wide scope. Legacy top-level `ebook_title`/`thumbnail_url`; `order_refundable_amount`; `open_refund_requests`; top-level `refund_request` is summary-only
   - GET /api/v1/user/orders/<id>/invoice/ (Bearer OR signed token) — downloads invoice PDF (Content-Disposition: attachment)
   - GET /api/v1/user/courses/enrolled/ — grouped catalog like GET /courses/; each enrolled book includes pdf_url as signed …/courses/<slug-or-by-id-N>/download/?token=… (not raw storage; empty slug uses ``by-id-<id>``)
   - GET /api/v1/user/courses/enrolled/<slug>/ | /enrolled/<id>/ — detail when enrolled; pdf_url signed download URL
   - GET /api/v1/user/courses/<slug>/download/ — serves enrolled full PDF (inline); Bearer OR ?token= (no Bearer). Append ``&meta=1`` for JSON ``download_url`` + ``expires_at`` only.

6) Binary tree (after qualifying MLM purchase)
   - GET /api/v1/user/tree/, /subtree/, /uplines/, /downlines/, /level/<n>/

7) Commission engine (Celery in prod; eager in tests)
   - Direct + upline + milestones; cap from SystemConfig; refunds reverse commissions.

8) Wallet, bands, withdrawals
   - Prefer GET /api/v1/user/earnings/ and GET /api/v1/user/payouts/ for dashboard bundles.
   - Band ladder unlocks sponsor-slot batches at configured indices.
   - Member POST withdraw; FINANCE approve/reject/export.

9) Sponsor slots — validate publicly; member lists batches; share logs channel.

10) Admin — password + OTP patterns; dashboards, users, reports (some stubs).


=============================================================================
DETAILED ENDPOINT REFERENCE
=============================================================================

Every subsection lists: METHOD URL | Auth | Request | Success data (inside envelope
unless noted) | Common errors.


MODULE: Authentication (/api/v1/auth/)
--------------------------------------------------------------------------------
POST /api/v1/auth/send-otp/
  Auth: None.
  Body (JSON): {
    "phone": "<optional when using email login; strict E.164 with + when supplied — e.g. +918589960592>",
    "email": "<optional>",
    "purpose": "LOGIN" | "KYC" | "ADMIN_LOGIN"
  }
  Rules: Provide phone (E.164 +…) and/or email. Bare domestic numbers without + are rejected. Not used for signup.
  Success data (when EXPOSE_OTP_IN_RESPONSE=true, default unless env disables):
    {
      "expires_in_seconds": 600,
      "otp": "<six-digit code; also INFO-logged under logger apps.authentication>"
    }
    When EXPOSE_OTP_IN_RESPONSE=false, "otp" is omitted (production SMS flows).
  message: "OTP sent"
  Errors: 429 — rate limited (max 3 sends / 10 minutes per identifier);
    errors may include {"detail": "rate_limited"}.

POST /api/v1/auth/register/send-otp/
  Auth: None.
  Body: {
    "phone": "<required; strict E.164 with leading +: e.g. +918589960592>",
    "email": "<optional; stored if provided>",
    "full_name": "<required>",
    "referral_code": "<required; e.g. Admin for company sponsor>"
  }
  Validates sponsor exists; rejects duplicate phone (or duplicate email when email supplied).
  Success: same envelope as send-otp (expires_in_seconds + otp when EXPOSE_OTP_IN_RESPONSE is on).
  Errors: 400 invalid referral / validation / duplicate user; 429 rate limit per phone.

POST /api/v1/auth/verify-otp-register/
  Auth: None.
  Body: {
    "phone": "<required; same E.164 as register/send-otp>",
    "otp_code": "6 chars"
  }
  Creates user from data stored on the matching unused REGISTER OTP (full_name, optional email, sponsor).
  Success data: {
    "user": { id, member_id, full_name, phone, email, referral_code, is_member, role },
    "tokens": { "access" },
    "is_book_purchased": <bool; true if the user has at least one PAID ebook order (single- or multi-item)>
  }
  Errors: 400 invalid OTP / expired session data / duplicate phone-email; note: complete within OTP TTL after send.

POST /api/v1/auth/verify-otp-login/
  Auth: None.
  Body (JSON): identifier — phone (strict E.164 with +, same as send-otp) or email — plus otp_code (6 chars).
  Success data: same user + tokens shape as register, plus "role" (session role) and the same is_book_purchased flag.
  Errors: 423 account locked (otp_locked_until); 404 user not found; 400 OTP errors.

POST /api/v1/auth/logout/
  Auth: Bearer access JWT recommended (session context); body optional.
  Success: data null; message "Logged out" — client removes stored access JWT.

GET/PATCH /api/v1/auth/me/
  Auth: Bearer.
  GET — Success data: user stub plus profile blocks for the profile screen:
    - Base user: id, member_id, full_name, phone, email, referral_code, is_member, role, is_book_purchased
      (same semantics as verify-otp-register / verify-otp-login: true if at least one PAID ebook order).
      referral_code is null until kyc_status is VERIFIED and MemberComplianceProfile exists on file.
    - personal_information: full_name, email_address, mobile_number, date_of_birth (DD/MM/YYYY), gender
    - member_information: member_id, joined_date (ISO), address, city, state, pin_code, country
    - display: profile_initial, avatar_url
    - account_status: account_status (+ label), kyc_status (+ label), pan_submitted, withdrawals_blocked,
      kyc_submission_allowed (bool), kyc_submission_mode ("instant"|"after_refund"),
      trigger_instant_kyc_submission (bool, mirrors SystemConfig),
      kyc_eligible_at (ISO datetime or null — earliest refund-window end when still blocked in after_refund mode),
      kyc_invitation_sent_at (ISO datetime or null; null in instant mode),
      tds_rate_percent (null until KYC approved), referral_link (null until KYC approved),
      referral_link_active (false until KYC approved)
    - feature_access: team_network, earnings, withdrawals, milestones, sponsor_slots, referral_program,
      compliance_submit (booleans). MLM sections and referral_program are true only when kyc_status is VERIFIED
      and MemberComplianceProfile exists. compliance_submit is true when submission is allowed and kyc is not VERIFIED.
    - tax_withholding: null until KYC approved; then { tds_rate_percent, reason }
    - withdrawal_band: null until KYC approved; then { current_band, bands }
    - earning_cap: null until KYC approved; then { limit, used, used_percent, remaining }
    - kyc_notice: { message, code } — code is refund_window | submit_kyc | pending_review | rejected | purchase_required | null
    - sponsor: { member_id, full_name } or null (visible before KYC approval — upline only)
    - binary_placement: null until KYC approved; then { position, level, position_label }
    - team_legs: null until KYC approved; then left_leg_count, right_leg_count, weaker_leg, leg labels, subtree_member_count
    - admin (only when caller is staff with role SUPER_ADMIN, FINANCE, or SUPPORT):
      - default_company_referral_code: active value used for sponsor resolution and reserved-code checks
        (SystemConfig override when set, otherwise DEFAULT_COMPANY_REFERRAL_CODE from the environment).
      - default_company_referral_code_environment: env/settings value before override (for display).
  PATCH — Edit profile fields (partial update).
    Content-Type: application/json
    Body fields:
      - full_name (string, optional): member display name.
      - email (string, optional): unique email for member login/profile.
      - date_of_birth (string, optional): DD/MM/YYYY (also accepts YYYY-MM-DD).
      - gender (string, optional): Male/Female/Other/Prefer not to say (or M/F/O/U).
      - address (string, optional): maps to compliance full_address.
      - city (string, optional)
      - pin_code (string, optional)
      - state (string, optional)
      - country (string, optional)
      - payout_preference (string, optional): "BANK" or "UPI".
      - default_company_referral_code (string, optional): SUPER_ADMIN only; max 64 chars after trim.
        Persists to SystemConfig; send empty string to clear override and use env DEFAULT_COMPANY_REFERRAL_CODE again.
        Same field is PATCHable on GET/PATCH /api/v1/admin/config/ (superadmin).
    Example body:
      {
        "full_name": "Unnikrishnan K",
        "email": "unnikrishnan@example.com",
        "date_of_birth": "17/08/1996",
        "gender": "Male",
        "address": "44, MG Road, Ernakulam",
        "city": "Kochi",
        "pin_code": "682001",
        "state": "Kerala",
        "country": "India",
        "payout_preference": "UPI"
      }
    Notes:
      - Send only the fields you want to update.
      - Name/email update on User; DOB/address fields update MemberComplianceProfile.
      - If neither field is sent, response still returns current profile payload.
      - If default_company_referral_code is present and the caller is not role SUPER_ADMIN: 403 errors.detail super_admin_only.
  Success data: same shape as GET; message "Updated"

POST /api/v1/auth/kyc/submit/
  Deprecated: returns 410 Gone — use POST /api/v1/auth/compliance/submit/.

GET /api/v1/auth/kyc/invite/
  Auth: None (public deep-link validator).
  Query: token (required) — signed payload from invitation email/SMS ({FRONTEND_BASE_URL}{KYC_INVITE_WEB_PATH}?kyc_token=...).
  Success data: {
    "valid": true,
    "kyc_submission_allowed": <bool>,
    "kyc_submission_mode": "instant"|"after_refund",
    "kyc_status": "PENDING"|"VERIFIED"|"REJECTED",
    "member_id": "<string>",
    "full_name": "<string>",
    "redirect_hint": "compliance"
  }
  Errors: 400 invalid_kyc_invite_token; 404 user not found.
  Note: member must still authenticate (OTP/JWT) to submit compliance; this endpoint only validates the link.

GET /api/v1/auth/kyc/status/
  Auth: Bearer.
  Success data includes: kyc_status, kyc_submission_allowed, kyc_submission_mode,
    kyc_submitted_at, kyc_reviewed_at, kyc_rejection_reason (or null),
    pan_last4, missing_acceptances (list of docs needing acceptance at latest version),
    has_compliance_profile, compliance_submission_version,
    compliance_submission (null if no profile, else: personal_details from User + address/DOB/gender;
    kyc_details (PAN/Aadhaar fields + verification_documents URLs); bank_details; nominee_details; updated_at).

POST /api/v1/auth/bank/
  Deprecated: returns 410 Gone — use POST /api/v1/auth/compliance/submit/.

POST /api/v1/auth/compliance/submit/
  Auth: Bearer. Content-Type: multipart/form-data (or multipart with JSON-compatible fields supported by parser).
  Text fields:
    date_of_birth (DD/MM/YYYY), gender (M|F|O|U),
    full_address, city, pin_code, state, country,
    pan_number, name_on_pan, aadhar_number (12 digits), name_on_aadhar,
    nominee_name, nominee_relationship, nominee_phone (strict E.164 +91 Indian mobile rule),
    nominee_date_of_birth (DD/MM/YYYY),
    account_holder_name, account_number, bank_name, ifsc, branch,
    account_type (SAVINGS|CURRENT|NRO|NRE|RECURRING|OTHER),
    optional payout_preference (BANK|UPI); omitted or blank defaults to UPI,
    optional upi_id for UPI payouts.
  Files: pan_document, aadhar_front, aadhar_back (PDF or image types as accepted by storage).
  Server syncs PAN + masked Aadhaar onto User alongside MemberComplianceProfile; sets kyc_status PENDING pending admin review.
  Success: { "kyc_status", "compliance_submission_version" }; 400 if mandatory agreements missing.
  Errors: 403 — { errors: { detail: "kyc_refund_window_active" } } when refund window still open (standard mode) or no PAID ebook purchase (instant mode).
  POST /api/v1/agreements/send-otp/ also returns 403 kyc_refund_window_active when submission is not yet allowed.

POST /api/v1/auth/validate-referral/
  Auth: None.
  Body: { "referral_code": "CODE" } or { "code": "CODE" }
  Success data: { "sponsor_name": "<full_name>" }
  Errors: 404 invalid code.


MODULE: Agreements (/api/v1/agreements/, member)
--------------------------------------------------------------------------------
GET /api/v1/agreements/
  Auth: Bearer.
  Query (optional): category (case-insensitive exact match on document category).
  Success: {
    "results": [
      { "id", "name", "category", "document_type", "year", "description",
        "content_html", "version", "pdf_url", "pdf_file_url", "requires_acceptance_for_compliance",
        "is_agreement_accepted" }
    ],
    "user": [
      {
        "id": <member user id>,
        "order_invoices": [
          { "order_id", "order_number", "invoice_number", "invoice_pdf_url", "paid_at" }
        ],
        "compliance_acceptance_proof": null | {
          "acceptance_batch_id": "<uuid>",
          "pdf_download_url": "<absolute URL with ?token=... signed query — paste in browser to download without Bearer>",
          "issued_at": "<ISO8601>",
          "verification": { "signature": "<hex>", "algo": "HMAC-SHA256" }
        }
      }
    ]
  }
  Notes: order_invoices lists PAID orders with GST invoice PDF URL when generated.
    compliance_acceptance_proof reflects the latest OTP acceptance batch; PDF is created lazily on first list/download.
    The downloaded proof PDF includes company logo (just200_logo.png), member declaration text from send-otp, attestation block (signed-by / date / reason / location), document table, and HMAC.

GET /api/v1/agreements/acceptance-proof/<acceptance_batch_id>/download/
  Auth (either):
    - Full URL as returned in user.compliance_acceptance_proof.pdf_download_url: includes query ``token=<signed payload>`` (valid ~7 days). No Authorization header required — for opening in a browser / WebView.
    - Or: Bearer access token for the same member (path batch must belong to that user).
  Path: acceptance_batch_id is a UUID (must match token when using token auth).
  Success: binary PDF stream; Content-Type application/pdf; Content-Disposition attachment.
  Errors: 401 missing/invalid/expired token and no Bearer; 404 if batch not found for owner or proof cannot be built.

GET /api/v1/agreements/compliance-legal/
  Auth: Bearer.
  No query params. Success data: { "results": [ ... ], "user": { "id": <int>, "full_name": "<member name>" } } — same object shape as each element of `results` from GET /api/v1/agreements/, but only rows where:
    is_active=true, requires_acceptance_for_compliance=true, and category is LEGAL DOCUMENT (case-insensitive).

POST /api/v1/agreements/send-otp/
  Auth: Bearer.
  Body (JSON): {
    "document_ids": [ <active doc ids that require acceptance> ],
    "declaration": "<required; non-empty trimmed text 1–4000 chars; shown on generated acceptance proof PDF>"
  }
  Uses member's stored phone/email only (never arbitrary numbers).
  OTP purpose AGREEMENT; OTP payload binds document_ids + user_id + declaration server-side.

POST /api/v1/agreements/verify/
  Auth: Bearer.
  Body (JSON): { "document_ids": [<same ids as OTP session — same order-independent match>], "otp_code": "......" }
  Success: acceptance_batch_id, accepted_document_ids; creates UserAgreementAcceptance rows and stores declaration for the batch.
  Errors: 400 if OTP session predates declaration field — request a new OTP via send-otp with declaration.


MODULE: Admin agreements & compliance (/api/v1/admin/…)
--------------------------------------------------------------------------------
GET /api/v1/admin/agreements/
  Auth: Bearer + IsSuperAdmin.
  Lists all LegalDocument rows (inactive included).
  Query (optional): category (case-insensitive exact match on document category).
    Note: accepts loose alias like "KYC and IDENTITY" and normalizes to "KYC & IDENTITY" before filtering.

POST /api/v1/admin/agreements/
  Auth: IsSuperAdmin.
  Body (JSON or multipart): name (required); category (required on create; must be one of:
    "LEGAL DOCUMENT", "KYC & IDENTITY", "PURCHASE & TAX" — case-insensitive, stored as canonical casing);
    document_type; year; description; content_html; version (default "1.0"); pdf_file (upload);
    requires_acceptance_for_compliance; is_active.
  At least one of content_html or pdf_file required on create/update.
  PATCH: category, when sent, must be one of the same three values (cannot be empty).

GET/PATCH/DELETE /api/v1/admin/agreements/<id>/
  Auth: IsSuperAdmin.
  PATCH partial update. DELETE is soft (is_active=false).
  If PATCH includes category, it must be one of: LEGAL DOCUMENT, KYC & IDENTITY, PURCHASE & TAX (same rules as POST).

GET /api/v1/admin/agreements/compliance-required/
  Auth: IsSuperAdmin.
  Returns the single active compliance-required LEGAL DOCUMENT agreement, or null data when none exists.
  Same field shape as admin agreement detail (LegalDocument fields; category and requires_acceptance_for_compliance are fixed by the server).

POST /api/v1/admin/agreements/compliance-required/
  Auth: IsSuperAdmin.
  Upsert singleton: creates on first call, updates the same row on subsequent POST/PUT.
  Body (JSON or multipart): same fields as POST /api/v1/admin/agreements/ except category and requires_acceptance_for_compliance are not accepted (forced to LEGAL DOCUMENT and true).
  Clears requires_acceptance_for_compliance on any other LegalDocument rows.
  At least one of content_html or pdf_file required on create/update.

PUT /api/v1/admin/agreements/compliance-required/
  Auth: IsSuperAdmin.
  Same upsert semantics as POST (full replace when row exists; create when missing).

PATCH /api/v1/admin/agreements/compliance-required/
  Auth: IsSuperAdmin.
  Partial update of the singleton; 404 if none exists yet.

DELETE /api/v1/admin/agreements/compliance-required/
  Auth: IsSuperAdmin.
  Soft-removes the singleton from compliance (is_active=false, requires_acceptance_for_compliance=false).

GET /api/v1/admin/compliance-queue/
  Auth: IsSupportAdmin.
  Returns rich pending submissions: profile fields, nominee, bank echoes, signed media URLs when DEBUG/media served.

POST /api/v1/admin/users/<id>/compliance/approve/
  Auth: IsSupportAdmin.
  Accepts optional body to approve single or bulk:
    - { "user_id": 123 }  OR  { "user_ids": [123, 456] }
  When body ids omitted, falls back to <id> in URL (backward compatible).
  Requires MemberComplianceProfile exists per user; sets kyc_status VERIFIED; kyc_reviewed_at now.
  For bulk, returns { approved_ids: [...], failed: [{id, reason}...] }.

POST /api/v1/admin/users/compliance/approve/
  Same as above, but without <id> in URL (preferred for bulk).

POST /api/v1/admin/users/<id>/compliance/reject/
  Auth: IsSupportAdmin. Body JSON { "reason": "..." } — sets REJECTED and stores reason.

POST /api/v1/admin/users/<id>/kyc/verify/
  Backward-compatible alias for compliance approve (still IsSupportAdmin).

GET /api/v1/admin/users/kyc-queue/
  Deprecated minimal list: user ids only — prefer GET /api/v1/admin/compliance-queue/.

POST /api/v1/admin/users/<id>/kyc/send-invitation/
  Auth: IsSupportAdmin.
  Manually send or resend the post-refund KYC invitation email/SMS (standard mode only).
  Body (JSON, all optional except user id when using bulk URL):
    - force (boolean, default false): resend even when kyc_invitation_sent_at is already set.
    - skip_refund_check (boolean, default false): allow send before any PAID ebook refund window has closed.
  When body omits user ids, uses <id> from URL.

MODULE: User — Earnings bundle (preferred for My Earnings UI)

GET /api/v1/user/earnings/
  Auth: Bearer.
  Query: include — optional. Comma-separated: "overview" (default when omitted),
    "ledger", or "overview,ledger" or "all" (both). Use "ledger" only to skip overview.
  Query: period — optional: today | 7d | 30d | fy | all (default all). Filters ledger.
  Query: type — optional: all | direct | passive | milestone | reversed | pending | withdrawal.
  Query: page (default 1), page_size (default 20, max 100) for the ledger table section.
    Pagination applies to `data.ledger.rows` only: `len(rows)` <= `page_size`, while
    `data.ledger.total_count` is the total count across the active `period` + `type` filters.
  Success data (subset keys omitted when include excludes that section):
    "summary": {
      "title", "subtitle", "lifetime_earnings",
      "cap": { used_pct, used, limit, remaining },
      "wallet": {
        available_to_withdraw, locked, withdrawn
      },
      "income": {
        "direct_l1" { amount, units, unit },
        "passive_l2_l4" { amount, units, unit },
        "milestone" { amount },
        "slots" { amount, redeemed, unit }     # `unit` is live SystemConfig.product_base_price
      },
      "band": { label, number, next, policy },
      "rule": { title, text },
      "fy_label", "kyc_status"
    },
    "filters": {
      "period", "type",
      "periods": [today, 7d, 30d, fy, all],
      "types": [all, direct, passive, milestone, pending, reversed, withdrawal]
    },
    "ledger": {
      "rows": [ unified rows ],
      "total_count", "page", "page_size"
    }
  Per-stream lifetime amounts are the four strings under summary.income.*.amount
  (direct_l1, passive_l2_l4, milestone, slots); summary.breakdown is not returned.
  Ledger row fields (commissions + milestones + withdrawals; aliases kept for existing clients):
    id, kind (COMMISSION|MILESTONE|WITHDRAWAL|WITHDRAWAL_REFUND), at (ISO datetime),
    date (e.g. "13 May 2026"), time (12h local, e.g. "01:45 PM"),
    type (Direct|Passive|Reversed|Pending; Milestone|Held|Pending for milestones;
      Withdrawal / Withdrawal Pending|Processing|Paid|Rejected|Failed; Withdrawal Refunded),
    detail, description (same string as detail, for table DESCRIPTION column),
    triggered_by { member_id, full_name }|null (joiner / source user; null for milestones),
    via_downline null (reserved for future tree context),
    level (L1–L4 for commissions; null for milestones),
    gross, tds, tds_deducted (same as tds), net, net_credited (same as net),
    status (CREDITED|REVERSED|PENDING|HELD), status_label ("Credited"|… title case for UI chips;
      slot-band-held credits get a " (Slot fund)" suffix on both label and description),
    source { member_id, full_name }|omitted (duplicate of triggered_by when present),
    referrals (int, milestone rows only),
    order_id (commissions, if present),
    slot_band_held (bool; true for COMMISSION and MILESTONE rows that landed while the
      recipient was inside a slot band 2/4/6/8 — see "Slot-band routing" note below),
    cash_credited (bool; true only when the row actually increased cash_balance — false for
      slot-band-held, REVERSED, PENDING, HELD rows and for withdrawal rows),
    withdrawal_id, payout_method, utr_number (withdrawal rows, when present),
    balance, running_balance (same decimal string; wallet cash balance immediately after this
    event when walking newest-first — see note below),
  Notes:
    - summary.wallet values are decimal strings. locked = cash not yet withdrawable under the wallet cooling
      window (length = SystemConfig.cooling_off_days, echoed as cooling_days on GET /api/v1/user/payouts/ and
      GET /api/v1/user/wallet/); available_to_withdraw matches withdrawal checks. For TDS and richer wallet fields
      use GET /api/v1/user/payouts/.
    - balance / running_balance are emitted per returned row in descending time order. The walk
      starts from Wallet.cash_balance and subtracts each row's net cash impact as it moves to older
      rows, so the value is the member's cash balance immediately after that line. Withdrawals
      appear as negative rows (kind WITHDRAWAL; gross and net equal -amount_requested, tds 0);
      status reflects the linked withdrawal request (Pending / Processing / Paid / Rejected /
      Failed). A rejected withdrawal also produces a WITHDRAWAL_REFUND row (positive gross/net)
      that restores the wallet. Older earnings rows are not rewritten when a new withdrawal is
      added — only newer rows reflect the reduced balance. REVERSED commission rows adjust the
      walk so totals stay consistent.
    - Withdrawal row TDS column is always 0; TDS on payouts is shown on GET /api/v1/user/payouts/.
    - Per-line tds may be 0 until engine writes tds_deducted.
    - Slots amount = count(REDEEMED sponsor codes issued to you) × SystemConfig.product_base_price.
      The per-code `unit` follows the live ebook base price — when admin changes the price the
      amount re-renders accordingly.
    - Slot-band routing: when wallet.current_band is 2, 4, 6 or 8 the band is a sponsor-slot
      band — direct, upline and milestone credits earned in that window count toward
      lifetime_earnings (total_earned, cap meter, band progression) but NOT toward cash_balance
      or available_to_withdraw. They are funded into the sponsor-slot pool instead. The ledger
      shows these rows with cash_credited=false, slot_band_held=true, and a status_label suffix
      "(Slot fund)"; their running_balance is unchanged from the next-newer row. Once
      total_earned crosses the band ceiling (e.g. ₹5000 from band 2), subsequent credits resume
      crediting cash automatically. Slot codes inside the band unlock progressively at equal
      steps (band 2: ₹4200, ₹4400, ₹4600, ₹4800, ₹5000 — analogous spacing for bands 4/6/8).
    - `page` and `page_size` are clamped server-side: `page` >= 1, `page_size` in [1..100].

POST /api/v1/admin/refunds/<id>/approve/
  Auth: IsFinanceAdmin.
  Flow:
    1) Pre-check: refund request exists, not closed, linked order is PAID.
    2) **Razorpay refund (GATEWAY only)** — before any DB row locks: calls Razorpay `payments/<payment_id>/refunds`
      with `amount` in paise (from `RefundRequest.amount`). Uses same credentials as checkout (`SystemConfig`
      or env). **DIRECT** payments (`razorpay_payment_id` prefixed `MANUAL-`): skip this step (offline settlement).
    3) Inside a transaction (row locks): re-verify open request + PAID order, then runs internal fulfilment —
      Razorpay partial amount is refunded; enrollments removed for the refunded line (or all enrollments when the request
      is order-wide with order_line_id null). Order stays **PAID** until cumulative **approved** refund amounts reach
      ``order.amount_paid`` (within a small rounding tolerance); then order is set **REFUNDED**, commissions are reversed
      once, and any remaining enrollments for that order are cleared. Sets request APPROVED plus `razorpay_refund_id`
      when the gateway returned one.
  Success: { id, status, approved_at, razorpay_refund_id | null }
  Errors:
    • 503 — Razorpay keys missing / client not configured (GATEWAY orders only; no DB changes).
    • 502 — Razorpay refund API error (safe message; order stays PAID; request stays open).
    • 400 — invalid state (closed / order not PAID / missing payment id on GATEWAY order).
    • 404 — request not found.
    • 500 — rare: Razorpay refund succeeded but finalize failed; audit `refund.razorpay_ok_db_failed` is written
      for ops; errors payload may include `razorpay_refund_id` for reconciliation.

GET/PATCH /api/v1/admin/config/
  Auth: IsSuperAdmin only.
  GET data: {
    "product_base_price", "gst_rate", "direct_commission", "upline_commission",
    "earning_cap" } as strings;

    "auto_placement_strategy": "LEFT_FIRST"|"RIGHT_FIRST"|"LONG_LEG"|"WEAK_LEG";
    "is_repurchase_commission_allowed" (boolean, default false) — when false, the
    commission engine does not distribute MLM commissions on a buyer's second and
    later PAID non-retail orders (first such order still commissions normally).
    "auto_process_milestone_bonuses" (boolean, default true) — when false, milestone
    bonuses are created as PENDING records and must be processed by admin via the
    Milestone Bonuses queue endpoints.
    "trigger_instant_kyc_submission" (boolean, default false) — when true, members may submit
    KYC/compliance immediately after a PAID ebook purchase (no refund-window wait; no invitation
    email/SMS). When false, submission opens after any PAID ebook order refund window closes and
    an invitation is sent (Celery beat, optional /auth/me safety net, or admin POST …/kyc/send-invitation/).
    Milestones (admin-configurable):
      - milestone_bonus_overrides: { "<threshold_int>": "<decimal|string|number>", ... }
        Example: { "10":"300", "25":"600", "50":"1000", "75":"1350", "100":"1600" }
      - milestones: [
          { "label":"T1", "index":1, "threshold":10, "bonus_gross":"300" },
          ...
        ]
        Note: `milestones` is computed from defaults with `milestone_bonus_overrides` applied.
    Razorpay (admin-configurable; secret is never returned):
      - razorpay: { "key_id": "<string|null>", "key_secret_set": true|false }
      - PATCH accepts: razorpay_key_id (string), razorpay_key_secret (string)

        }

      - default_company_referral_code: active value (DB override or env).
      - default_company_referral_code_environment: env/settings value only.
      - default_company_referral_code_override: stored override string or null when empty (env used).
